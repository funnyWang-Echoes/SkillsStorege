# Close Reading Spec

精读文档的唯一格式来源。每段精读块长什么样、图表怎么嵌入、代码怎么放，都在这里。

## 段落精读块

**默认一段原文 = 一个精读块。** 按论文原文顺序排列，不重排。

允许合并的情况：2-3 段原文如果**逻辑紧密**（如同一机制的连续描述、同一实验的设置+结果+分析），可以合并成一个精读块，但要在标题标注段落范围：`### §3.2 段落 2-3（合并）`。合并时翻译仍逐段给出，分析写一次。

如果原文一段太长（超过 300 字），可以在同一精读块内分段翻译。如果太短（仅一句），与下段合并。

### 标题格式

```markdown
### §3.2 段落 2
```

- 章节号 + 段落号，不加 Section Title
- 可选括号摘要不超过 10 字：`### §3.2 段落 2（router 设计）`
- 不要写 `### §3.2 · Router Design · 段落 2（router 接收特征点和全局潜码...）` 这种长标题

### 精读块内容

```markdown
### §3.2 段落 2

**中文翻译：**
为了忠实解码采样随机性和文本条件到最终合成，
我们的 router 在做决策时考虑了文本集成的全局潜码。
具体来说，router 接收中间特征点和全局潜码作为输入，
输出对每个 expert 的选择概率。

**即时分析：**
- **这段在说什么**：router 不只看特征点，还看全局信息（文本条件）来选 expert
- **【呼应 §2.1】**：Intro 说 "GAN 难以 scale up"，这里用 sparse MoE 回应
- **【回应 Q3】**：§3.1 问 router 怎么选 expert，这里回答了。Q3 已回答 ✓
- **有没有问题**：router 的具体架构还没说清楚

**疑问 Q5：** router 用什么架构算选择概率？→ 待 §3.3 回答

![Figure 2: Router 架构图](../mineru/raw/images/figure_2.jpg)
*图 2：Router 接收特征点和全局潜码，输出 expert 选择概率。*
```

### 字段说明

| 字段 | 必须性 | 说明 |
|---|---|---|
| 章节定位 | 必须 | `### §X.Y 段落 N`，可选括号摘要 ≤10 字 |
| 中文翻译 | 必须 | 完整翻译这段原文的意思，不是短摘。**翻译是纯翻译，不混入分析/批评/审计。** |
| 即时分析-这段在说什么 | 必须 | 一句话概括 |
| 即时分析-怎么连接前段 | 有连接时必须 | 写 `【呼应 §X.Y】` 或 `【回应 Q#】` 或 `【新话题】` |
| 即时分析-有没有问题 | 有问题时必须 | 逻辑跳步、证据不足、定义含糊等。**审计/批评只放这里，不混入翻译。** |
| 疑问 | 有疑问时产生 | `**疑问 Q#:** ...`，后面的段落回答时写 `【回应 Q#】... Q# 已回答 ✓` |
| 论文图表 | 段落有对应图表时必须 | 见下方"图表嵌入" |

### 翻译原则

- 翻译是再创作，保留原文技术准确性和论证逻辑
- **翻译区只放翻译。分析、批评、审计、代码引用全部放"即时分析"区，不混入翻译。**
- 术语第一次出现时给中英对照，如"mixture-of-experts (MoE, 混合专家)"
- 公式不翻译，用 KaTeX 原样保留

### 分析原则

- 分析长在段落旁边，不是另起一节
- "怎么连接前段"是核心——这是阅读链的灵魂
- 疑问有 ID 追踪，三态闭环：✓ 已回答 / ◐ 部分回答 / ✗ 未回答
- "有没有问题"要诚实——逻辑跳步、未定义符号、实验设置可疑，当场标出来
- 正文不贴代码——代码在文末"核心代码附录"集中展示

### 代码未公开的处理

如果论文代码未公开：
1. 在 Method 章节开头（§3.0 或第一个 Method 段落）说一次"代码未公开"
2. 后续不要每段重复"代码未公开"
3. 只在发现具体缺失的实现细节时标注，如"attention 类型论文未明说，代码未公开无法核对"
4. 不创建代码附录章节
5. 把复现性缺口在疑问追踪表和举一反三中显式讨论

## 图表嵌入

MinerU 解析后的图片在 `mineru/raw/` 目录下。精读文档中**必须**把论文的 Figure 和 Table 插入到对应段落的精读块里。

### 写法

```markdown
![Figure 2: Router 架构图](../mineru/raw/images/figure_2.jpg)

*图 2：Router 接收特征点和全局潜码，输出 expert 选择概率。*
```

### 规则

- 图片路径用相对路径，从 `output/` 目录指向 `mineru/raw/` 下的图片
- 每张图下面必须有图注（斜体），说明这张图在读什么、对应论文哪段
- **Table 优先用 Markdown 表格重建关键数字**，不要直接用截图。只在表格太复杂无法重建时才用 MinerU 截图。
- 多子图（如 Figure 5a/5b/5c）可以分别嵌入，各自加图注
- HTML 生成器会把本地图片转成 base64 内嵌

## 代码处理

正文**不贴代码**。读到 Method 的机制时，在即时分析里只写一句引用。代码集中放在文末"核心代码附录"。

### 核心代码附录格式

```markdown
## 核心代码附录

代码来源：https://github.com/zhujiapeng/Aurora

### 代码块 1: Router 实现

**对应论文：** §3.2 Router Design
**文件：** `models/text_generator.py:1655-1690`

```python
class Router(nn.Module):
    ...
```

**讲解：**
- `self.gate` 对应论文的 router 线性层
- 代码用了 `top_k=2`，论文写的是完全稀疏——实现差异
- `global_w` 就是论文说的"文本集成的全局潜码"
```

### 规则

- 每个代码块标注：对应论文章节、文件路径:行号
- 代码块后必须有"讲解"：实现什么、对应论文哪个公式/机制、和论文描述有什么差异
- 只贴核心代码（router、loss、关键 forward 逻辑），不要贴全部源码
- 每段代码不超过 40 行；太长就截取关键部分，用 `# ...` 标注省略
- `deep` 档必须有代码附录；`full` 档有公开代码时也应有
- **代码未公开时不创建此章节**

## 阅读导航

在元信息后面、正文前面，加一个简洁的导航：

```markdown
## 阅读导航

| Section | 段落数 | 疑问 |
|---|---|---|
| Abstract | 1 | Q1-Q3 |
| §1 Introduction | 5 | Q1-Q5 |
| §3 Method | 4 | Q5-Q8 |
| §4 Experiments | 5 | Q8-Q11 |
| §A Appendix | 3 | Q4 理论闭合 |
| 疑问追踪表 | — | 11 个（7✓ 1◐ 3✗） |
| 举一反三 | — | 4 条 |
| 代码附录 | 6 块 | — |
```

## 附录精读深度

- **理论附录**（如数学推导、ELBO 分析）：必须完整精读，和正文 Method 同等深度
- **实现附录**（如训练超参、模型配置）：可以浓缩为 1-2 个精读块，但必须列出复现所需的关键参数（学习率、batch size、训练步数、loss 权重等）
- **额外实验附录**（如更多可视化、额外 ablation）：可以合并为 1 个精读块概要，标出关键发现

## 举一反三

每条必须给出：迁移到什么任务/场景、为什么能迁移（来源段落）、最小验证实验是什么。

- **机制迁移**：...（来源：§3.2 + §4.1）。最小验证：在 X 任务上试 Y，看 Z 是否改善。
- **方法改进**：...（来源：§3.3 的局限）。最小验证：去掉/替换 W，看 gFID 变化。

## 一句话总结

三句话说清楚：在干什么、做得怎么样、值不值得跟。

## 核心代码附录（有公开代码时）
（代码未公开时跳过此章节）
```

## 举一反三原则

- 每条必须指明来源段落
- 每条必须给出具体迁移路径：迁移到什么任务/场景、为什么能迁移
- 每条必须给出最小验证实验：怎么做最小实验来验证这个迁移是否可行
- 不要写"这个思路理论上可以迁移到 X"这种无法操作的抽象建议

## 不需要的东西

- 没有 reading card 数量配额
- 没有 evidence embed 配额
- 没有 mechanism block 字数要求
- 没有 contribution claims 表格
- 没有 argument-map / assumption-ledger / evidence-matrix 等中间 notes
- 没有复杂的 deep-report-check.json
- 正文不贴代码（代码在文末附录）

质量由阅读深度本身决定。如果每段都有翻译 + 分析 + 图表 + 呼应，疑问都有追踪，代码附录有讲解，那就是好的精读。
