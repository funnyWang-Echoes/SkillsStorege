# Research Progress 写作助手（中文版）

目标：把零散的实验记录、结果表格、周工作流水，整理成**结构完整、量化、诚实、低 AI 味**的进展文档。

本 skill 必须可独立使用：所有必要模板、写作原则和检查清单都在本文件与 `references/`、`templates/` 中，不依赖任何外部 API、模型或浏览器。即使用户只触发 `research-progress`，也必须能完成「判别文体 → 收集素材 → 套模板 → 生成 → 自检」全流程。

---

## 0. 一上来先判别文体

收到写作请求后，先用一句话确认用户要的是哪一类（若用户已明确则可跳过）：

**A. 科研进展类**（偏研究内容本身）
- 探索性实验小结 / 实验归档
- 月度 / 季度科研月报（给导师 / 委员会 / 基金）
- 文献分析 / 文献调查报告
- 综合研究报告 / Benchmark 总结报告

**B. 工作进展类**（偏事务与时间线）
- 科研周报 / 工作周报
- 项目进展汇报（给主管 / 甲方 / 跨团队）

不确定时，用 AskUserQuestion 让用户二选一，并顺便问清：读者是谁（导师 / 委员会 / 甲方 / 组内）、周期（周 / 月 / 季度）、字数或页数要求。

## 1. 收集素材（不要凭空编）

生成前先向用户要原始材料。**不要编造数据、指标、文献**。若材料不足，列出缺什么，而不是用套话填补。

科研进展类需要：
- 实验设置（数据来源、方法/模型、基线、评估指标、时间段）
- 结果（表格、关键数字、对比口径）
- 遇到的问题 / 被排除的假设
- 相关文献（若有）

工作进展类需要：
- 本周 / 本周期做了什么（尽量原始的流水记录即可）
- 关键产出（数字、交付物、PR/MR 链接）
- 遇到的问题与解决
- 下周计划

如果用户只给了一段流水账，照常处理——本 skill 的价值就是把流水账提炼成结构化文档。

## 2. 写作原则（两类通用，核心质量要求）

这些原则内化自高质量科研写作范例（详见 `references/writing-principles.md`），务必在生成时遵守：

1. **数据驱动，量化优先**：每个结论尽量有数字或表格支撑。避免「取得了一定进展」「效果较好」这类空话；用 `RMSE 0.685 vs 基线 0.707（-3.15%）` 这种写法。
2. **诚实报告 negative result**：失败的实验、被排除的假设同样有价值，要写明「排除了什么、为什么不行」。这往往比报喜更有科研价值。
3. **公平基线 + 统计显著性**：对比必须是同口径；优先用置信区间 / 多次 seed 均值，而非单点数字；明确 baseline 是否公平。
4. **机制分析重于堆叠结果**：不只报数字，要解释 *为什么*（失败/成功的根因、归纳偏置是否匹配、数据或信息量的瓶颈）。
5. **结构清晰、可追溯**：背景 → 动机 → 方法/数据 → 结果 → 机制 → 结论 → 讨论（相关工作 / 局限）→ 文献。

**去 AI 味硬规则（两类都要套）**：
- 温度设低（0.3–0.5），逻辑更严密，少煽情修饰。
- 单项汇报控制在 150 字以内（数据/统计密集型内容可升级为子节并嵌入表格，不受此限）。
- 强制具象化：必须出现具体参数（如温度 250℃、精度 98.5%、耗时 12.4s、RMSE 0.685）。
- 禁用「值得注意的是」「正如我们所知」「总而言之」「综上所述」等 AI 高频过渡词；用分段和图标符号（✅ ❌ ⚠️ ★）代替。
- 涉及未发表公式 / 核心代码 / 专利，用「算法 A」「化合物 B」代替，保护科研隐私。

| 文体 | 模板文件 | 关键结构 |

| 月度 / 季度科研月报 | `templates/科研月报-导师.md` | 摘要 → 本月进展 → 关键结果表 → Blocker+解决方案 → 下月逐周计划 → 向导师提问 → 论文/投稿状态 |

- `7.7探索性实验小结.md`：三线实验完整归档的典范，结构可直接套。

## 4. 工作进展类 —— 模板与用法

按文体从 `templates/` 取对应模板：

| 文体 | 模板文件 | 关键结构 |
|---|---|---|
| 科研周报 / 工作周报 | `templates/工作周报.md` | 本周工作进展（分点+数据）→ 关键成果与价值 → 遇到的主要问题（客观）→ 解决方案 & 下周计划 |
| 项目进展汇报 | `templates/项目进展汇报.md` | 项目背景与目标 → 执行过程与关键节点 → 成果与数据（vs 计划）→ 经验教训 → 风险与后续 |

工作周报写法借鉴现有进展类 Skill（LobeHub `progress-report` 的 Monthly Advisor Report 结构）与成熟的周报 Prompt 范式，要点：
- **本周进展分点罗列，用数据说话**：常规点每条自成一段、不超过 150 字；数据/统计密集型工作（文献审计、benchmark 汇总、轨迹归因）可升级为三级子节并嵌入统计表。
- **遇到的主要问题客观描述**，不用情绪化词汇；每个问题配一个解决方案。
- **下周计划只明确要做什么**：用条目列出具体任务，不排周次、不列可交付物、不写数字计数。
- **关键成果与价值按场景选 2–3 类量化**：科研场景用技术 / 决策 / 工程·团队，产业场景用业务 / 技术 / 团队，用百分比或绝对值。

详见 `references/weekly-report-guide.md`。

## 5. 生成后的自检清单（交付前必过）

逐条检查，不满足就改，不要交付带壳的文档：

- [ ] 没有「取得了一定进展」「效果较好」之类无数字支撑的空话
- [ ] 每个关键结论都有具体数值或表格
- [ ] negative result / blocker 被如实写出，没有粉饰
- [ ] 对比是同口径、注明 baseline 是否公平（科研类）
- [ ] 没有出现「值得注意的是」「综上所述」等 AI 套话
- [ ] 读者能从文档直接看懂：做了什么、结果如何、为什么、下一步
- [ ] 字数 / 页数符合用户要求（月报 1–2 页、委员会 3–5 页等）

## 6. 交付格式

- 默认产出一份 Markdown 文档（含标题、元信息、结构化章节、表格）。
