# 周报 / 工作汇报写法

本文件用于 `research-progress` skill 生成「工作进展类」文档。写法借鉴现有进展类 Skill（LobeHub `progress-report` 的 Monthly Advisor Report 结构）与成熟的周报 Prompt 范式——因为周报 / 工作记录没有特别好的现成文档，优先复用经过验证的 Skill 结构与 Prompt。

---

## 一、核心结构（周报）

```
# 工作周报 — [姓名] — [起始日期] 至 [结束日期]

## 一、本周工作进展
1. [任务] — [做了什么，数据/交付物]，[量化结果]
2. [任务] — ...
（常规点每条 ≤150 字；数据/统计密集型工作可升级为 ### 子节并嵌入统计表）

## 二、关键成果与价值
（按场景选 2–3 类：科研=技术/决策/工程·团队；产业=业务/技术/团队，量化优先）
- 技术价值：引入/验证 [方案]，指标提升 [X%]
- 决策价值：纠正 [判断] / 调整 [路线]
- 工程·团队价值：沉淀 [工具/文档/流程]

## 三、遇到的主要问题
- [问题1]：[客观描述，无情绪词] → [已采取的解决 / 待解决]
- [问题2]：...

## 四、解决方案 & 下周计划
- [任务1]：说明要做什么，不写具体数字、不列可交付物、不排周次
- [任务2]：...
- [任务3]：...

（下周计划只明确要做什么，不用排周一周二、不列可交付物）

```

## 二、核心结构（项目进展汇报）

```
# [项目名称] 进展汇报 — [日期]

## 一、项目背景与目标
为什么要做；预期达到什么效果

## 二、执行过程与关键节点
时间线（甘特式描述）；遇到的挑战与解决方案；协作亮点

## 三、成果与数据
实际完成情况（vs 计划）；量化成果（数据对比）；用户/客户反馈

## 四、经验教训
做得好的（可复用）；需要改进的（避免再犯）；给类似项目的建议

## 五、风险与后续
当前风险；后续维护 / 下一步计划
```

## 三、量化与去 AI 味

- **突出技术难点和解决方案**：不要写成流水账。「本周修复了 bug，开发了功能」是差的；「修复第三方库版本冲突导致的登录卡顿，页面加载从 3.2s 优化至 0.8s（提升 75%）」是好的。
- **体现对产品和团队的价值**：这个改动帮了谁、省了什么。
- **适当用量化表达**：百分比、绝对值、耗时、错误率。
- **数据/统计密集型工作**：文献审计、benchmark 汇总、轨迹归因等可升级为三级子节并嵌入统计表，不强压 150 字；统计引用权威真源，区分「任务可定义性」与「复现包率/成功率」。
- **禁用 AI 套话**：「值得注意的是」「综上所述」「值得一提的是」。用分段和图标代替。
- **温度 0.3–0.5**，逻辑严密，少用煽情修饰。

## 四、可直接抄的 Prompt（科研周报）

```
你是一位严谨的 [方向] 研究者 / 工程师，擅长逻辑梳理。
请根据我提供的碎片化工作记录，生成一份结构完整的周报。

要求：
- 本周工作进展：分点罗列，用数据说话（如精度 98.5%、耗时 12.4s）。
- 遇到的主要问题：客观描述，不用情绪化词汇。
- 解决方案 & 下周计划：列出要做的任务，只说明要做什么，不排周次、不列可交付物、不堆数字。
- 单条 ≤ 150 字；禁用「值得注意的是」等过渡词；出现具体参数。

输入源数据：
[在此粘贴本周零散记录 / 代码报错 / 实验日志]
```

> 隐私提示：涉及未发表公式、核心代码、专利时，用「算法 A」「模块 B」代替。
